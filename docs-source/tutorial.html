🎯 設計哲學

OpenSpec 的設計建立在四大原則之上：

🌊

Fluid（流暢）

沒有 phase gate，透過 DAG 依賴圖自動推導「下一步該做什麼」，讓使用者在準備好時自然推進。

🔄

Iterative（迭代）

每個 Artifact 可獨立產生、修改、重新產生，不需要推倒重來整個流程。

✨

Easy（簡單）

零配置即可開始（內建 spec-driven schema），進階使用者可逐步自訂 schema、context、rules。

🏗️

Brownfield-first（棕地優先）

透過 Delta Specs 機制，支援在既有規格上做增量變更，而非要求從零撰寫完整規格。

💡 一句話總結 OpenSpec 把「AI 幫你寫 code 之前應該產出的文件」拆成可插拔的骨架零件（artifacts），用可替換的骨架定義（schema）組裝起來，確保 AI 在動手前跟你對齊。

🧩 核心概念

五大核心元素

📋

Spec（規格）

描述系統行為的文件，含 Requirements 和 Scenarios。存放在 specs/，是 Source of Truth。

📝

Change（變更提案）

對系統的一次修改，打包為獨立資料夾，含所有 Artifacts。存放在 changes/。

🧱

Artifact（產出物）

Change 中的文件零件 — proposal、specs、design、tasks。由 Schema 定義有哪些、什麼順序。

🔀

Delta Spec（差異規格）

描述 ADDED / MODIFIED / REMOVED 的規格變更，Archive 時合併回 Source of Truth。

📦

Archive（歸檔）

完成 Change 後，Delta Specs 合併到 specs/，Change 移入 archive/ 目錄。

兩大區域

🟢 specs/（永久規格區）

Source of Truth — 記錄「系統現在應該做什麼」

只在 Archive 時透過 Delta Specs 合併更新，隨著每次 Archive 而成長。

🟡 changes/（變更提案區）

臨時工作區 — 每個變更一個資料夾，含所有 Artifacts

完成後歸檔至 archive/。

🔗 DAG 依賴圖

OpenSpec 將工作流建模為有向無環圖（DAG）。每個 Artifact 透過 requires 宣告依賴，系統自動推導建置順序。

內建 spec-driven 依賴圖

proposal
↙      ↘
specs
design
↘      ↙
tasks
↓
apply（實作）

DAG 運作機制

系統使用 Kahn's 演算法進行拓撲排序，提供三個核心查詢：

方法功能
getNextArtifacts(completed)返回所有依賴已滿足、可以開始的 Artifact
getBlocked(completed)返回尚在等待依賴的 Artifact 及其未完成的依賴
getBuildOrder()返回完整的拓撲排序建置順序
📌 完成偵測 透過檔案系統：如果 generates 指定的檔案存在（支援 glob 匹配），該 Artifact 即為「已完成」。
💡 Dependencies 是 enablers，不是 gates 它們表示「什麼可以做」，不是「什麼必須做」。你可以跳過 design 如果不需要。specs 和 design 可以平行建立（都只依賴 proposal）。

🧬 AI Prompt 組裝機制

當使用者執行 openspec instructions <artifact> 時，系統將五層資訊組裝成結構化的 XML Prompt：

五層 Prompt 組裝
Layer 1: project_context — 來自 config.yaml → context（專案背景）
Layer 2: rules — 來自 config.yaml → rules[artifactId]（per-artifact 約束條件）
Layer 3: dependencies — 來自 schema.yaml → artifact.requires（依賴檔案路徑與狀態）
Layer 4: template — 來自 schemas/<name>/templates/（輸出結構骨架）
Layer 5: instruction — 來自 schema.yaml → artifact.instruction（AI 產出指引）

組裝後的 Prompt 結構

<artifact id="proposal" change="add-2fa" schema="spec-driven">

  <task>Create the proposal artifact for change "add-2fa".</task>

  <project_context>
    這是一個使用 TypeScript 的專案...
  </project_context>

  <rules>
    - 必須包含 rollback 章節
  </rules>

  <dependencies>
    <dependency id="proposal" status="done">
      <path>openspec/changes/add-2fa/proposal.md</path>
    </dependency>
  </dependencies>

  <template>
    # Proposal: {{change-name}}
    ...
  </template>

  <instruction>
    Create the proposal document...
  </instruction>

  <unlocks>Completing this artifact enables: specs, design</unlocks>

</artifact>

📋 Proposal (proposal.md) — 為什麼 + 做什麼 
# Proposal: Add Dark Mode

## Intent
Users have requested a dark mode option to reduce eye strain.

## Scope
In scope:
- Theme toggle in settings
- System preference detection

Out of scope:
- Custom color themes (future work)

## Approach
Use CSS custom properties + React context.

## Rollback Plan
Revert the theme context and CSS changes.

## Risks
| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Flash of unstyled content | Medium | Low | Use prefers-color-scheme |

何時更新：scope 變了、intent 更清楚了、approach 根本性改變

描述「什麼在變」，不是整份 spec 重寫。使用 ADDED / MODIFIED / REMOVED 語義。

詳見 Delta Specs 章節。

Spec 格式規範

# Auth Specification

## Requirements

### Requirement: User Authentication
The system SHALL issue a JWT token upon successful login.

#### Scenario: Valid credentials
- GIVEN a user with valid credentials
- WHEN the user submits login form
- THEN a JWT token is returned
- AND the user is redirected to dashboard

#### Scenario: Invalid credentials
- GIVEN invalid credentials
- WHEN the user submits login form
- THEN an error message is displayed
📌 RFC 2119 關鍵字 MUST / SHALL — 絕對要求｜SHOULD — 建議，有例外｜MAY — 選用
Spec 是行為契約，不是實作計畫：寫可觀察的行為、input/output、error condition，不寫 class name、library 選擇。

🔀 Delta Specs（差異規格）

Delta Specs 是 OpenSpec 支援 Brownfield 開發的核心機制。變更提案中的規格檔案描述的是與現有規格的差異。

四種操作語義

操作語法說明
ADDED## ADDED Requirements新增需求，Archive 時插入至目標規格
MODIFIED## MODIFIED Requirements修改既有需求，必須包含完整替換內容
REMOVED## REMOVED Requirements移除需求，必須包含 Reason 與 Migration
RENAMED## RENAMED Requirements重新命名，使用 FROM: / TO: 格式

Delta Spec 範例

## ADDED Requirements

### Requirement: User can enable 2FA
The system SHALL allow users to enable two-factor authentication.

#### Scenario: Enable 2FA
- WHEN user navigates to security settings
- THEN system displays 2FA setup wizard

## MODIFIED Requirements

### Requirement: User login
The system SHALL require 2FA verification after password authentication
when 2FA is enabled on the account.

## REMOVED Requirements

### Requirement: Security question verification
**Reason**: Replaced by 2FA
**Migration**: Users will be prompted to set up 2FA on next login

Archive 時的合併順序

1. RENAMED 重新命名 key
→
2. REMOVED 從映射中刪除
→
3. MODIFIED 完整替換
→
4. ADDED 插入（已存在則報錯）

為什麼用 Delta

增量描述

只需描述「什麼變了」，不需重寫整份規格

衝突可見

MODIFIED 和 REMOVED 明確標示受影響需求

可審查

Delta 格式讓 reviewer 一眼看出變更範圍

可逆

Archive 前隨時可調整 Delta 內容

⚙️ Project Config

openspec/config.yaml 是最重要的客製化入口，控制 AI 生成 artifacts 的品質。

三個設定項

# openspec/config.yaml

# ① schema — 預設使用的骨架定義
schema: spec-driven

# ② context — 注入到「所有」artifact 的 AI prompt
context: |
  ## Tech Stack
  - Language: Java 17
  - Framework: Spring Boot 3.x
  - Database: PostgreSQL 15 + Flyway
  - API Style: RESTful, OpenAPI 3.0

  ## Architecture
  - Layered: Controller → Service → Repository
  - DDD for core business logic

  ## Conventions
  - Package: com.company.project.{domain}.{layer}
  - Integration tests use Testcontainers

# ③ rules — 針對「特定」artifact 的規則
rules:
  proposal:
    - 必須包含 rollback plan
    - 必須識別受影響的微服務
  specs:
    - 使用 Given/When/Then 格式
    - 必須包含 error scenario
  design:
    - 必須包含 sequence diagram（Mermaid）
    - 必須說明 DB schema 變更
  tasks:
    - 每個 task 必須可在 4 小時內完成
    - 必須包含測試的 task

差異比較

設定項注入範圍用途建議
schema—預設 schema建議設定
context所有 artifacts描述技術棧、架構、慣例強烈建議
rules.<artifact>只有對應 artifact針對性生成規則視團隊需求
⚠️ Rules 的 key 必須對應 schema 中的 artifact id 內建 spec-driven 的 id 為：proposal、specs、design、tasks。Custom schema 要用你自己定義的 id。

🔧 Schema 系統

Schema = 可替換的骨架定義，決定你的 workflow 有哪些 artifacts、什麼順序組裝。

Schema 解析優先順序

Schema 名稱的來源（Change 層級）

1. --schema CLI flag（最高優先）
2. .openspec.yaml 中的 schema 欄位
3. config.yaml 中的 schema 欄位
4. 內建 fallback：spec-driven（最低優先）

Schema 檔案的查找位置（三層）

1. 專案本地：openspec/schemas/<name>/（最高優先）
2. 使用者全域：$XDG_DATA_HOME/openspec/schemas/
3. 套件內建：<package>/schemas/（最低優先）

Schema 結構

openspec/schemas/my-workflow/
├── schema.yaml          # Artifact 定義 + Apply 設定
└── templates/           # Markdown 模板
    ├── proposal.md
    ├── spec.md
    ├── design.md
    └── tasks.md

schema.yaml 完整欄位

欄位必填說明
nameYesSchema 名稱，kebab-case
versionYes版本號
descriptionNo描述文字
artifactsYesArtifact 定義陣列
artifacts[].idYes唯一識別碼
artifacts[].generatesYes輸出檔名，支援 glob
artifacts[].templateYestemplates/ 下的模板檔
artifacts[].instructionNoAI 生成指示
artifacts[].requiresYes依賴的 artifact id 陣列
apply.requiresYes進入 apply 前必須完成的 artifacts
apply.tracksYes追蹤實作進度的檔案

🛠️ Custom Schema 建立教學

決策：Fork vs 從零建立

Fork（推薦）

跟 spec-driven 差異小，只想改 template 或加一個 artifact。

openspec schema fork spec-driven my-workflow

從零建立

完全不同的流程。

# 互動式
openspec schema init my-workflow

# 非互動式
openspec schema init rapid \
  --description "Rapid iteration" \
  --artifacts "proposal,tasks" \
  --default

完整建立流程

建立 Schema（Fork 或 Init）

選擇適合的方式建立基本結構

撰寫 Templates

# openspec/config.yaml

config.yaml 的 rules key 必須對應 schema.yaml 中的 artifact id

🔌 Plugin 整合

🔑 關鍵理解 OpenSpec 不呼叫 Plugin，OpenSpec 產出的文件（design.md、tasks.md）裡「寫著」要 AI 去呼叫哪些 Plugin 的 slash command。整合是透過 AI prompt instruction 實現的「文字層串接」，不是程式碼層的 API 呼叫。

兩個專案的角色

📐 OpenSpec — 編排者

管「做事的順序」— 透過 Schema 定義 Artifact DAG，決定先產 proposal、再寫 spec、再出 design、最後列 tasks。

🔧 Plugin — 執行者

管「怎麼做事」— 提供具體的 slash command（/scaffold-jpa、/generate-sd 等），AI 呼叫後自動生成程式碼。

整合架構

┌─────────────────────────────────────────────┐
│              你的 Spring Boot 專案             │
│                                               │
│  openspec/           ← OpenSpec 管「流程」     │
│  ├── config.yaml                              │
│  ├── schemas/spring-boot-dev/                 │
│  └── changes/add-2fa/                         │
│      ├── design.md  ← 裡面「寫」要用哪些       │
│      │                 plugin skills          │
│      └── tasks.md   ← 裡面「列」要跑哪些       │
│                        slash commands         │
│                                               │
│  .claude/            ← Plugin 管「執行」       │
│  ├── commands/                                │
│  │   ├── scaffold-jpa.md     ← 來自 plugin   │
│  │   ├── generate-sd.md      ← 來自 plugin   │
│  │   └── gen-api-task.md     ← 來自 plugin   │
│  └── rules/coding-standards.md               │
└─────────────────────────────────────────────┘

串接靠 AI 理解自然語言

Schema 的 instruction 欄位用自然語言告訴 AI「到這一步請呼叫 /scaffold-jpa」，AI 讀懂後自行呼叫。不需要任何程式碼整合。

  pull_request:

      - 'openspec/**'

  validate:
    runs-on: ubuntu-latest

      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4

🔧 Troubleshooting

問題原因解法
Change not found未指定名稱指定：/opsx:apply add-dark-mode；openspec list 確認
No artifacts ready前置 artifact 未完成openspec status --change <name> 看什麼 blocked
Schema not found拼字錯誤或路徑不對openspec schemas 列出所有
Commands not recognized設定檔未更新openspec init + openspec update；重啟 AI 工具
Artifacts 品質差context/rules 不夠具體在 config.yaml 加詳細的 context 和 rules
Template not foundschema.yaml template 欄位指向不存在的檔案openspec schema validate
Rules 沒生效rules key 跟 artifact id 不一致確認 key 對應 schema 中的 artifacts[].id
Circular dependencyA requires B, B requires Aopenspec schema validate 會偵測

設定 Custom Schema 前的 Checklist

  • config.yaml — schema 設為 custom schema 名稱
  • config.yaml — context 描述了技術棧和架構
  • config.yaml — rules 針對每個 artifact 設了規則
  • schema.yaml — 定義了所有需要的 artifacts
  • schema.yaml — 依賴關係正確（無循環）
  • schema.yaml — apply.requires 和 apply.tracks 正確
  • templates/ — 每個 artifact 都有對應的模板檔
  • openspec schema validate 通過
  • openspec instructions 確認 prompt 正確
