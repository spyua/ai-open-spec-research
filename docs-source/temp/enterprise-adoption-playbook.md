# OpenSpec 企業導入統籌手冊

> 文件性質：High-Level 統籌指南
> 目標讀者：負責推動導入的技術負責人（你）
> 前置文件：[企業導入分析報告](enterprise-integration-analysis.md)

---

## 你的角色定位

你不是在「導入一個工具」，你是在**建立一個跨部門的規格治理體系**。工具只是載體。

你要解決的根本問題：

> 各部門的 AI 開發工具鏈各自成熟，但「做什麼」和「做到什麼程度算對」沒有共同語言。

你的成功不是「每個團隊都裝了 OpenSpec」，而是：

> 任何人打開任何專案的 `openspec/specs/`，都能理解這個系統的行為契約。

## 第一部分：統籌框架

### 1.1 三層治理模型

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   集團層（你統籌）                                       │
│                                                         │
│   ┌───────────────────────────────────────────────────┐ │
│   │  定義：                                            │ │
│   │  · enterprise-standard schema（基礎流程）          │ │
│   │  · 規格品質標準（什麼算合格的 Spec）                │ │
│   │  · 跨部門規格引用規範                               │ │
│   │  · 導入節奏與里程碑                                 │ │
│   │                                                    │ │
│   │  不定義：                                           │ │
│   │  · 各部門用什麼 Plugin                              │ │
│   │  · 各部門的 Design artifact 長什麼樣                │ │
│   │  · 各部門的 tasks.md 怎麼寫                         │ │
│   └───────────────────────────────────────────────────┘ │
│                         │                               │
│                    fork + 擴展                           │
│                         │                               │
│   ┌─────────┐  ┌───────┴───┐  ┌─────────┐  ┌────────┐ │
│   │ 後端部門 │  │ 前端部門   │  │ 資料部門 │  │ Mobile │ │
│   │         │  │           │  │         │  │        │ │
│   │ 自訂：   │  │ 自訂：     │  │ 自訂：   │  │ 自訂： │ │
│   │ ·schema │  │ ·schema   │  │ ·schema │  │ ·schema│ │
│   │ ·config │  │ ·config   │  │ ·config │  │ ·config│ │
│   │ ·plugin │  │ ·plugin   │  │ ·plugin │  │ ·plugin│ │
│   │  整合    │  │  整合      │  │  整合    │  │  整合  │ │
│   └─────────┘  └───────────┘  └─────────┘  └────────┘ │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**你管什麼，不管什麼：**

| 你管 | 你不管 |
|------|--------|
| 所有部門的 Proposal 和 Spec 必須存在 | Proposal 和 Spec 的具體內容由各部門自行判斷 |
| Spec 必須用 Given/When/Then 格式 | Design artifact 的結構各部門自訂 |
| 跨部門影響的變更必須走完整流程 | 部門內的小修小補可以用 rapid schema |
| enterprise-standard schema 的演進 | 各部門 schema 的日常維護 |
| 導入進度的追蹤與障礙排除 | 具體的 Plugin 技術整合 |

### 1.2 你要做的五件事

```
 1. 定義標準     →  enterprise-standard schema + 規格品質基準
 2. 選對試點     →  找到最有痛點且最有意願的團隊
 3. 證明價值     →  用一個真實案例展示 before/after
 4. 建立機制     →  讓規格管理變成流程的一部分，而非額外負擔
 5. 擴散推廣     →  從試點經驗萃取 pattern，讓其他團隊複製
```

## 第二部分：導入前的關鍵決策

### 2.1 決策一：規格的邊界在哪裡？

這是最重要的決策。不是所有變更都值得寫規格。

**建議的分級制度：**

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│   Level 0: 不需要 OpenSpec                                │
│   ─────────────────────                                  │
│   · Bug fix（原因明確、修復範圍小）                        │
│   · 文件更新、typo 修正                                   │
│   · 依賴升級（無行為變更）                                 │
│   · 純重構（不改變外部行為）                               │
│                                                          │
│   Level 1: 輕量流程（rapid schema）                       │
│   ─────────────────────                                  │
│   · 單一 API 新增/修改                                    │
│   · 部門內的功能調整                                      │
│   · 不影響其他團隊的變更                                   │
│   · 產出：proposal.md + tasks.md                          │
│                                                          │
│   Level 2: 標準流程（enterprise-standard schema）         │
│   ─────────────────────                                  │
│   · 新功能開發                                            │
│   · 跨服務的行為變更                                      │
│   · API 契約變更（會影響其他團隊）                          │
│   · 產出：proposal + specs + design + tasks               │
│                                                          │
│   Level 3: 強化流程（enterprise-standard + 額外審查）      │
│   ─────────────────────                                  │
│   · 安全性相關變更                                        │
│   · 資料模型重大異動                                      │
│   · 跨部門協作的大型功能                                   │
│   · 產出：完整流程 + 跨部門 Spec Review                    │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**你需要決定的：** 每個 Level 的判定標準是什麼？誰有權決定一個變更屬於哪個 Level？

**建議：** 起步時只區分 Level 0 和 Level 2。規則越簡單，採用阻力越小。

### 2.2 決策二：enterprise-standard schema 包含什麼？

這是集團層級的共識，需要各部門 Lead 一起決定。

**最小可行 Schema（建議起步）：**

```yaml
name: enterprise-standard
version: 1
artifacts:
  # 文件型 — 所有部門共用
  - id: proposal
    generates: proposal.md
    requires: []
  - id: specs
    generates: specs/**/*.md
    requires: [proposal]

  # Skill 執行型 — 各部門自訂
  # （部門 fork 後在此插入自己的 Skill 執行型 artifact）

  # 文件型 — 實作記錄
  - id: implementation
    generates: implementation.md
    requires: [specs]
```

**不要一開始就加太多東西。** 基礎 Schema 只需要文件型 artifact（proposal、specs、implementation），Skill 執行型 artifact 由各部門 fork 後自行加入。

**Proposal 模板的統一要素（建議）：**

```markdown
## 變更動機
<!-- 為什麼要做這件事？連結需求單號 -->

## 影響範圍
<!-- 哪些服務？哪些團隊？ -->

## 風險評估
<!-- 安全性、效能、相容性、資料遷移 -->

## 成功標準
<!-- 怎麼知道這件事做完了、做對了？ -->
```

**Spec 模板的統一要素（建議）：**

```markdown
### Requirement: <名稱>
<行為描述，使用 MUST/SHOULD/MAY>

#### Scenario: <場景名稱>
- GIVEN <前置條件>
- WHEN <觸發動作>
- THEN <預期結果>
```

**Design 和 Tasks 的模板不需要統一。** 各部門的技術棧不同，設計文件的結構差異很大，強制統一只會製造摩擦。

### 2.3 決策三：規格的審查機制

Spec 寫了但沒人看，等於沒寫。你需要決定：

| 問題 | 選項 | 建議 |
|------|------|------|
| 誰審查 Spec？ | A. 部門內 Lead / B. 跨部門 Review / C. 架構師 | 起步用 A，跨部門變更用 B |
| 何時審查？ | A. Spec 完成後 / B. 納入 PR Review / C. 獨立 Review 流程 | B，融入現有流程阻力最小 |
| 審查什麼？ | A. 格式合規 / B. 行為正確性 / C. 完整性 | 起步只看 A+C，正確性由 verify 輔助 |
| 不合格怎麼辦？ | A. 退回修改 / B. 記錄但放行 / C. 阻擋合併 | 起步用 B，成熟後升級為 A |

## 第三部分：試點執行手冊

### 3.1 選擇試點團隊的標準

**必要條件：**
- [ ] 團隊 Lead 認同「規格管理」的價值（不需要全員認同）
- [ ] 團隊已有成熟的 Plugin Flow（證明執行層不是問題）
- [ ] 近期有一個中等規模的新功能要開發（不是 bug fix，不是巨型重構）

**加分條件：**
- [ ] 團隊最近因為「需求理解不一致」踩過坑
- [ ] 團隊的功能會影響其他部門（跨部門價值容易展示）
- [ ] 團隊 Lead 願意分享試點經驗

**避免：**
- 選正在趕 deadline 的團隊
- 選對任何新流程都抗拒的團隊
- 選功能太小（看不出價值）或太大（風險太高）的案例

### 3.2 試點的三週節奏

```
Week 1: 準備
├── Day 1-2: 與團隊 Lead 1:1，對齊期望
│   · 明確說明：這不是要取代你們的 Plugin
│   · 明確說明：只加一層上游規格，不改現有流程
│   · 一起選定試點的功能案例
│
├── Day 3: 環境設定
│   · npm install -g @fission-ai/openspec
│   · openspec init（在試點專案中）
│   · 建立/部署 enterprise-standard schema
│   · 設定 config.yaml（注入團隊的技術棧 context）
│
└── Day 4-5: 團隊簡報（30 分鐘以內）
    · 展示 OpenSpec 的核心概念（5 分鐘）
    · 展示完整流程 demo（15 分鐘）
    · Q&A（10 分鐘）
    · 不要講理論，直接 demo

Week 2: 實戰
├── 團隊用 OpenSpec 開始試點功能
│   · openspec new change <feature-name>
│   · 寫 proposal → specs → design → tasks
│   · 按 tasks.md 用 Plugin 執行
│   · openspec verify → openspec archive
│
├── 你的角色：觀察者 + 即時支援
│   · 不要替他們寫 Spec
│   · 記錄他們卡住的地方
│   · 記錄他們覺得多餘的步驟
│   · 記錄他們覺得有價值的時刻
│
└── 每日 5 分鐘 check-in
    · 今天用 OpenSpec 順不順？
    · 有沒有覺得多餘的步驟？
    · 有沒有「還好有寫 Spec」的時刻？

Week 3: 回顧與調整
├── Day 1-2: 收集回饋
│   · 與團隊成員 1:1（不只是 Lead）
│   · 問題清單：什麼有用、什麼沒用、什麼該改
│
├── Day 3: 調整 Schema
│   · 根據回饋調整 enterprise-standard schema
│   · 根據回饋調整模板
│   · 決定是否需要 rapid schema
│
└── Day 4-5: 撰寫試點報告
    · Before/After 對比
    · 具體的數據（時間、返工次數）
    · 團隊的直接引述
    · 建議的調整事項
```

### 3.3 試點的成功 / 失敗判斷

**成功信號：**
- 團隊成員說「還好有寫 Spec，不然這個邊界案例會漏掉」
- PR Review 時有人引用 Spec 來討論
- 團隊主動問「下一個功能也用 OpenSpec 嗎？」

**失敗信號：**
- 團隊把 Spec 當作「交作業」，寫完就沒人看
- 所有 Spec 都是 AI 生成、人類沒有審查就直接用
- 團隊覺得「多了一倍的文件工作，但沒有減少任何溝通」

**失敗時的應對：**
不要硬推。回頭檢視：

1. 是不是選錯了功能案例？（太小看不出價值）
2. 是不是 Schema 太重？（減少 artifact 數量）
3. 是不是團隊本來就溝通順暢，不需要規格層？（那就換一個有痛點的團隊）

### 4.1 推廣的四個階段

          採用率
            │
       100% ┤                                          ┌──────
            │                                     ┌────┘
        80% ┤                                ┌────┘
            │                           ┌────┘

            │                 ┌────┘
        40% ┤            ┌────┘

         0% ┤
            └──┬────────┬────────┬────────┬────────── 時間
              試點     擴散     標準化    深度整合
             1-2月    2-3月    2-3月     持續

Phase 1     Phase 2    Phase 3    Phase 4
試點        擴散       標準化      深度整合

### 4.2 Phase 2 擴散：你要做的事

試點成功後，不要一次推給所有團隊。用「漣漪式擴散」：

**Step 1：讓試點團隊分享經驗**

- 安排一次 30 分鐘的團隊分享（不是你講，是試點團隊講）
- 重點不是工具操作，而是「我們遇到什麼問題、OpenSpec 怎麼幫到我們」
- 真實案例 > 理論說明

**Step 2：招募第二批團隊**

- 從分享會後主動表達興趣的團隊中選
- 每個團隊指派一個「OpenSpec Champion」——不是你，是團隊內部的人
- 你的角色從「執行者」轉為「教練」

**Step 3：建立部門級 Schema**

每個新加入的團隊都需要：

```bash
# 從集團標準 fork
openspec schema fork enterprise-standard <department>-workflow

# 客製化
# 編輯 openspec/schemas/<department>-workflow/schema.yaml
# 加入部門特有的 artifact（如 api-design, component-design 等）

# 設定預設
# 在 openspec/config.yaml 設定 schema: <department>-workflow
```

### 4.3 Phase 3 標準化：建立制度

當多數團隊都在用 OpenSpec 時，需要建立制度化的機制：

**制度 1：規格審查納入 PR 流程**

```
PR Checklist:
  □ 程式碼 Review 通過
  □ 測試通過
  □ openspec verify 通過          ← 新增
  □ Spec 變更已被相關團隊確認      ← 新增（跨部門變更時）
```

（注意：不需要在 PR Checklist 中列出具體的 Skill 指令或 tasks.md 項目。Skill 執行型 artifact 的追蹤由 Schema 的 DAG 依賴和 `openspec verify` 自動處理。）

**制度 2：定期的規格健康檢查**

每月或每季，檢查：
- 有多少功能有對應的 Spec？
- 有多少 Spec 在 archive 中（代表完成了完整生命週期）？
- 有多少 Spec 被其他部門引用？
- 有多少次「因為有 Spec 所以提前發現問題」？

**制度 3：Schema 治理**

```
Schema 變更流程：

集團層級 (enterprise-standard)
└── 變更需要：架構師提案 → 各部門 Lead 討論 → 共識後更新
    · 頻率：每季最多一次
    · 原則：只加不改、向後相容

部門層級 (<department>-workflow)
└── 變更需要：部門 Lead 決定 → 通知架構師
    · 頻率：不限
    · 原則：不可移除 enterprise-standard 中的必要 artifact
```

### 4.4 Phase 4 深度整合：長期願景

```
                     目前可做                     未來可做
              ┌─────────────────┐        ┌──────────────────┐
              │                 │        │                  │
              │  OpenSpec 產出   │        │  OpenSpec 產出    │
              │  artifact 文件   │        │  artifact 文件    │
              │       │         │        │       │          │
              │       ▼         │        │       ▼          │
              │  人工閱讀        │        │  Plugin 自動讀取  │
              │  tasks.md       │        │  tasks.md        │
              │       │         │        │       │          │
              │       ▼         │        │       ▼          │
              │  人工執行        │        │  自動觸發         │
              │  Plugin 指令     │        │  Plugin 指令      │
              │                 │        │       │          │
              └─────────────────┘        │       ▼          │
                                         │  自動驗收         │
                                         │  openspec verify │
                                         │       │          │
                                         │       ▼          │
                                         │  行為規格驅動     │
                                         │  測試案例生成     │
                                         │                  │
                                         └──────────────────┘
```

深度整合需要的技術投資：
1. Plugin Skill 增加「讀取 OpenSpec artifact」的能力
2. OpenSpec 增加 Hook 機制（apply 階段前後觸發外部指令）
3. Spec 的 Given/When/Then → 自動生成測試骨架

**建議：Phase 4 不要在導入初期規劃。** 先讓 Phase 1-3 跑穩，Phase 4 的需求會自然浮現。

## 第五部分：常見阻力與應對話術

### 5.1 來自團隊的阻力

**「我們已經有 Plugin Flow 了，為什麼還需要這個？」**

> 你們的 Plugin 解決的是「怎麼做」——生成程式碼、確保品質。OpenSpec 解決的是「做什麼」——行為規格。兩者是不同層級的東西。就像你們有 Checkstyle 不代表不需要寫需求文件。

**「寫 Spec 太花時間了，我直接寫 code 比較快。」**

> 寫 code 確實比較快，但改 code 很慢。上次因為需求理解不一致返工花了多少時間？Spec 的成本是前置的，但它避免的是後期的返工成本。而且 OpenSpec 支援 AI 生成 Spec，你只需要審查，不需要從頭寫。

**「我們團隊溝通很順暢，不需要額外文件。」**

> 如果只看你們團隊內部，可能確實不需要。但當你們的 API 變更影響到前端團隊時呢？當新人加入需要理解系統行為時呢？Spec 的價值不只是當下的溝通，還有未來的可追溯性。

**「又一個要學的工具…」**

> OpenSpec 的學習成本很低：4 個 CLI 指令（new、verify、archive、status）就能跑完流程。而且它不改變你們現有的 Plugin 用法，只是在上游加一層。

### 5.2 來自管理層的阻力

**「ROI 是什麼？」**

> 直接 ROI 很難量化，但可以用代理指標：
> - 因需求不一致導致的返工次數（目標：下降 30%）
> - 跨部門溝通的會議時間（目標：減少，因為有文件可以異步 review）
> - 新人理解系統行為的時間（目標：可以直接讀 Spec，不需要問人）

**「會不會拖慢開發速度？」**

> 短期會有學習曲線。但小變更可以用 rapid schema（只要 proposal + tasks），不是每個改動都要走完整流程。而且 Spec 是 AI 輔助生成的，不是純手工。

**「為什麼不用 Confluence / Notion / Google Docs？」**

> 三個原因：
> 1. OpenSpec 的 Spec 是版控的（跟程式碼在同一個 repo），Confluence 不是
> 2. OpenSpec 有結構化的 Delta Spec 機制，可以追蹤行為變更，文件編輯器做不到
> 3. OpenSpec 的 Spec 可以被 AI 工具直接消費（作為程式碼生成和驗證的輸入），純文件不行

## 第六部分：你的每週行動清單

### 導入前（2 週）

```
Week -2:
  □ 完成 enterprise-standard schema 草案
  □ 與 2-3 個部門 Lead 一對一溝通，收集意見
  □ 調整 schema 草案

Week -1:
  □ 確定試點團隊和試點功能
  □ 在試點專案中設定 OpenSpec 環境
  □ 準備 30 分鐘的團隊 Demo
```

### 試點期（3 週）

```
Week 1:
  □ 完成團隊 Demo
  □ 團隊開始用 OpenSpec 走試點功能
  □ 每日 5 分鐘 check-in

Week 2:
  □ 觀察團隊使用狀況
  □ 記錄痛點和亮點
  □ 必要時即時調整（模板、rules）

Week 3:
  □ 收集回饋（1:1 訪談）
  □ 調整 schema 和模板
  □ 撰寫試點報告
```

### 擴散期（每個新團隊）

```
每個新團隊加入：
  □ 指定團隊內的 OpenSpec Champion
  □ Champion 參加 1 小時的 hands-on workshop
  □ 協助團隊 fork schema 並客製
  □ 第一個功能全程陪跑
  □ 第二個功能起放手，只做 check-in
```

## 第七部分：衡量你是否成功

### 領先指標（過程指標，提前反映趨勢）

| 指標 | 衡量方式 | 健康值 |
|------|---------|--------|
| 團隊主動使用率 | 新功能有建立 OpenSpec change 的比例 | > 60% |
| Spec 審查率 | Spec 在 PR 中被 review 的比例 | > 50% |
| Champion 活躍度 | 各團隊 Champion 是否持續推動 | 每月至少 1 次分享 |

### 落後指標（結果指標，需要時間才能觀察）

| 指標 | 衡量方式 | 目標 |
|------|---------|------|
| 返工率 | 因需求不一致的返工次數 / 總功能數 | 下降 30% |
| 跨部門溝通效率 | 跨部門功能的平均交付時間 | 下降 20% |
| 規格累積 | `openspec/specs/` 中的 spec 數量 | 持續成長 |
| 歸檔完成率 | 已 archive 的 change / 已建立的 change | > 70% |

### 你應該擔心的紅旗

- **Spec 都是 copy-paste** → 模板太死板，需要調整
- **只有 Lead 在寫 Spec，其他人不碰** → 採用沒有滲透到團隊層級
- **Change 建立了但從不 archive** → 流程跑了一半就放棄，可能太重
- **沒有任何跨部門引用** → 價值停留在部門內，沒有實現跨部門目標
- **Schema 分裂嚴重** → 各部門的 schema 差異太大，統一感消失
